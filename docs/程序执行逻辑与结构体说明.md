# 程序执行逻辑与结构体说明

## 1. 主执行逻辑

1. 程序入口在 src/main.rs。
2. 先读取两套数据模板配置：
   - 工序模板：data_config.json
   - 品质模板：品质检验.json
3. 再准备两套数据库配置路径：
   - 工序数据库：database_config.json
   - 品质数据库：quality_database_config.json
4. 将 6 个 Excel 文件组织为任务列表（3 个工序 + 3 个品质）。
5. 对每个任务独立构建并运行一条 Pipeline：
   - Reader：ExcelReader
   - Creater：WorkRecordCreater
   - Sink 链：DataChecker -> SqliteDatabaseSink -> StatsFlowSink -> EndSinkType
   - 其中 StatsFlowSink 内部按配置 `config/stats/sink_flow.json` 的 `sink_order` 顺序执行统计流程（`puaration`、`operator_puaration`、`material_production`）。
6. ExtractorPipeline 运行时分三段并发：
   - 读线程：逐行读取 Excel 原始行并送入缓冲
   - 创建线程：按配置把行数据聚合为 Data 对象
   - Sink 线程：消费 Data，校验、写库、统计并落盘
7. 每个任务处理完后，相关 sink 在 Drop 时完成缓冲刷新与统计结果持久化。

## 2. 数据流视角

1. ExcelReader 读取行数据，首行作为表头。
2. WorkRecordCreater 使用表头索引将多行项目记录聚合成一条业务记录 Data。
3. DataChecker 先做完整性检查，再执行几何规则检查，给 Data 打有效/无效标记。
4. SqliteDatabaseSink 根据 Data.is_valid() 分流到有效表或无效表。
5. StatsFlowSink 根据配置顺序调度具体统计 sink。
6. PuarationSink 从有效记录提取工艺向量，计算每个物料的向量纯度统计。
7. OperatorPuarationSink 从有效记录和备注信息中提取主机手，统计操机手维度的向量纯度。
8. MaterialProductionSink 从有效记录统计每个物料的去重批号次数。
9. EndSinkType 作为链尾空实现，保证链路闭合。

## 4. 与当前需求直接相关的关键结构体

1. Data
   - 控制字段 ready 与记录 valid 状态。
   - 决定记录进入有效表还是无效表。
2. WorkRecordCreater
   - 决定如何按索引键聚合记录。
   - 当前已支持索引别名（如物料品号/物料品名）。
3. DataChecker + GeometryCheckRule
   - 实现数据逻辑约束校验。
4. SqliteDatabaseSink
   - 执行有效/无效分表落库。
5. PuarationSink
   - 产出物料工艺向量纯度相关统计。
6. OperatorPuarationSink
   - 产出操机手维度的工艺向量纯度统计。
7. MaterialProductionSink
   - 产出每个物料的去重批号生产次数统计。

## 5. 运行产物

1. 主数据库（按任务配置）
   - 工序数据：database_config.json 指向的 SQLite 文件
   - 品质数据：quality_database_config.json 指向的 SQLite 文件
2. 纯度统计
   - JSON：data/puaration_stats.json
   - SQLite：puaration_sink_config.json + puaration_database_config.json 指定
3. 物料生产次数统计
   - JSON：data/material_production_stats.json
   - SQLite：data/material_production_stats.db
4. 操机手纯度统计
   - JSON：data/operator_puaration_stats.json
   - SQLite：data/operator_puaration_stats.db

## 6. 设计理念

1. 配置驱动（Configuration-Driven）
   - 数据模板、数据库列、统计输出列均通过 JSON 配置驱动，减少硬编码。
   - 目标是让新增 Excel 口径和客户字段变化更多通过改配置完成。
2. 流水线分层（Pipeline Layering）
   - 读取、聚合、校验、落库、统计分层解耦。
   - 每层只关注单一转换职责，便于替换和增量扩展。
3. 有效性优先（Validity First）
   - 先校验再入库，明确区分有效/无效记录。
   - 统计模块仅消费有效记录，避免污染分析结果。
4. 扩展优先于重写（Extension over Rewrite）
   - 在既有链路通过新增 sink 的方式扩展统计能力（纯度、生产次数）。
   - 不破坏主链路前提下逐步演进。
5. 可观测与可追踪（Observability）
   - 通过日志与进度表输出，让批处理过程可追踪、可定位问题。

## 7. 设计原则与设计模式

### 7.1 设计原则

1. 单一职责原则（SRP）
   - ExcelReader 只负责读取。
   - WorkRecordCreater 只负责把行数据聚合成 Data。
   - DataChecker 只负责校验。
   - SqliteDatabaseSink 只负责数据库落库。
   - PuarationSink 与 MaterialProductionSink 只负责各自统计。
2. 开闭原则（OCP）
   - 通过新增 sink、新增 rule 扩展行为，不改已有主流程结构。
   - 新客户需求优先通过新模块与新配置接入。
3. 依赖倒置（DIP）
   - ExtractorPipeline 依赖 DataReader、DataCreater、DataSink 抽象 trait。
   - 具体实现（ExcelReader、WorkRecordCreater、各 Sink）可替换。
4. 组合优于继承
   - 采用 sink 组合链路而非继承层次扩展功能。
   - 通过 next_sink 逐层转发形成可扩展处理管道。
5. 最小知识原则（LoD）
   - 上层只和必要接口交互，不直接依赖下游实现细节。

### 7.2 已使用的设计模式

1. 管道/责任链模式（Pipeline + Chain of Responsibility）
   - DataChecker -> SqliteDatabaseSink -> StatsFlowSink -> EndSinkType。
   - StatsFlowSink 内部再按配置执行各统计步骤。
   - 每个节点处理后转发给下一节点。
2. 策略模式（Strategy）
   - DataCheckRule 是校验策略抽象，GeometryCheckRule 是具体策略实现。
3. 模板数据原型模式（Prototype）
   - Data::from_template 通过克隆模板快速生成新业务记录。
4. 配置对象模式（Configuration Object）
   - ConfigReader 将 JSON 配置解析为结构体对象并驱动运行。
5. 生产者-消费者并发模式（Producer-Consumer）
   - ExtractorPipeline 使用通道与线程分离读取、创建、消费阶段。

### 7.3 模式带来的收益

1. 新统计口径可通过新增 sink 接入，改动局部化。
2. 新校验规则可通过新增 DataCheckRule 实现接入。
3. 新 Excel 口径可通过配置和别名映射适配。
4. 并发流水线提升大文件处理吞吐。
